## Overview

- Priority: P1
- Status: Complete
- Goal: make the external browser preview a better docs review surface with secure TOC/sidebar navigation, status, diagnostics, copy, and export feedback.

## Key Insights

- Browser is the real product surface today.
- Docs teams need status, diagnostics, reliable copy/export, and a fast generated TOC/sidebar more than risky MPE power features.
- Existing payload metadata can be reused before adding new indexing systems.
- TOC should be generated from sanitized heading metadata/rendered headings, not from executable parser hooks.

## Requirements

- Add visible render/connection/copy/export status improvements.
- Improve diagnostics clarity while preserving last good preview.
- Add an autogenerated TOC/sidebar that can sit left or right, collapse on small screens, and jump to headings.
- Add heading anchor/copy-link affordances if they can be implemented without mutating source Markdown.
- Add optional outline/metadata display using existing payload data.
- Prevent layout shift and token leakage.

## Architecture

```text
server RenderPayload(metadata.toc) -> browser preview renderer -> TOC/sidebar + toolbar/status/diagnostics -> copy/export actions
```

Payload changes should be optional and backward-compatible.

## Implementation Steps

1. Inventory existing diagnostics and toolbar states.
2. Design minimal status model: connected, rendering, updated, stale, copy ok/fail, export ok/fail.
3. Add accessible status UI with stable dimensions.
4. Build TOC/sidebar from `metadata.toc` when available; fall back to sanitized rendered headings if needed.
5. Add left/right placement option stored in browser local storage only; do not send preferences to server.
6. Add collapsible mobile behavior and keyboard-accessible heading jumps.
7. Add heading anchor/copy-link only if copied links use the local preview URL without tokens or use fragment-only links.
8. Keep preview updates scroll-preserving and non-destructive on render error.
9. Expand copy/export tests for token stripping, TOC exclusion/inclusion policy, and user feedback states.

## Success Criteria

- User sees whether preview is connected, current, failed, copied, or exported.
- User can navigate headings from a left or right TOC/sidebar.
- TOC works after saved-file refresh without losing scroll or expanding layout unpredictably.
- Render failures preserve the last good preview.
- Copy/export status is visible and non-noisy.
- No tokens appear in DOM fragments used for copy/export.

## Risk Assessment

- Risk: browser UI grows into a complex app.
- Mitigation: status and TOC/sidebar only; no graph, no heavy settings UI, no routing.
- Risk: layout shift on refresh hurts review flow.
- Mitigation: stable toolbar/status dimensions and scroll-preserving updates.

## Security Considerations

- Browser consumes sanitized payloads only.
- Keep CSP model intact.
- Sanitizers remain authoritative for copy/export.
- TOC and heading anchors must not include preview/session tokens.
- No inline scripts, remote fetches, or execution surfaces.
